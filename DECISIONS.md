## ADR-001 · SQLite + WAL 模式

**日期**：2026-07-13
**状态**：已采纳
**阶段**：P0

**背景**：项目是个人/小团队本地工具，单机运行，并发要求低（≤10 并发用户），数据规模预估万级题目。需要零部署依赖、文件级备份友好。

**方案**：使用 SQLite 3，启用 WAL（Write-Ahead Logging）模式，PRAGMA foreign_keys = ON。

**考虑过的替代**：
- PostgreSQL：需要部署服务，单机场景过重
- JSON 文件：无并发、无事务，初期简单但 P3 统计阶段必然重写
- DuckDB：分析友好但应用层事务与并发不成熟

**后果**：
- 优势：单文件备份、零依赖、文件复制即迁移
- 代价：写并发受 WAL 锁限制（acceptable，~1000 写/秒）
- 后续如需多机部署，需迁移到 PostgreSQL（参见 ADR-002 数据层抽象边界）

## ADR-002 · 题目 ID 用字符串而非自增整型

**日期**：2026-07-13
**状态**：已采纳
**阶段**：P0

**背景**：题目需要承载"年份-来源-序号"语义信息，便于在 wiki/导出的 Markdown 文件名、URL、组卷题单中直接识别。

**方案**：`M{4位年份}-{大写来源缩写}-{序号}`，例如 `M2024-NCZK-001`。应用层用 Pydantic pattern 校验。

**考虑过的替代**：
- 自增 INTEGER 主键：纯技术 ID，无业务含义，跨数据迁移困难
- UUID：无业务含义，太长不便阅读

**后果**：
- 优势：可读、跨系统兼容（Markdown 链接、文件名、URL 都自然）
- 代价：必须维护"年份-来源-序号"分配的全局约定（人工管理或 P4 录入引擎自动）
- 应用层强制校验可发现手误（如小写 `nczk` 或 2 位年份 `24`）

## ADR-003 · 提示词外部化（prompts/*.txt）

**日期**：2026-07-13
**状态**：已采纳
**阶段**：P0（占位）/ P4（实际填充）

**背景**：OCR 与 LLM 提示词迭代频繁，硬编码在代码里会导致改提示词要改代码、发版。需求方是教师（非工程师），希望"打开 txt 文件改一改就能试"。

**方案**：所有提示词存为 `prompts/<name>.txt`，通过 `app.services.prompt_loader.load_prompt(name)` 加载。Loader 用 mtime 缓存，文件变化时自动重新读取（热重载）。

**考虑过的替代**：
- 提示词入数据库：读写 UI 友好但跟代码版本脱钩
- YAML 配置文件：但提示词常含 JSON 示例和自然语言模板，YAML 引用与转义不友好

**后果**：
- 优势：git diff 自然、教师可改无需发版、hot-reload 支持实验
- 代价：缺失文件会被 `load_prompt` 抛 FileNotFoundError（明确优于静默 fallback）

## ADR-004 · 原子写入：临时文件 + fsync + os.replace

**日期**：2026-07-13
**状态**：已采纳
**阶段**：P0

**背景**：导出的 YAML / Markdown / HTML 文件若写一半被中断（断电、OOM），会留下半成品污染输出。备份数据同样不能写一半。

**方案**：`atomic_write_text(path, content)` 先写 `.path.tmp` → `flush()` + `os.fsync(fd)` → `os.replace(tmp, path)`。失败时清理 tmp 文件。

**数据库层**用 SQLite 的 `Connection.backup()` API 做热备份。

**考虑过的替代**：
- 直接 write：风险高
- 写完后 rename：跨平台 rename 语义差异（Windows 不允许覆盖）

**后果**：
- 优势：写入要么完成要么没发生
- 代价：写小文件比直写多 1 次 fsync 成本（acceptable）

## ADR-005 · AI 是建议者，不是决策者

**日期**：2026-07-13
**状态**：已采纳
**阶段**：P0（原则）/ P4（落地）

**背景**：AI 在 OCR 文本识别、元数据推断、答案生成上可能出错。直接写库会让错误沉淀。

**方案**：AI 输出只在 UI 上展示给用户，由用户决定"采纳 / 跳过 / 修改"。数据库只接受用户确认后的数据。审查界面（P4）按题号滑动校对，左栏原始 PDF、右栏 OCR 文本 + 元数据建议 + 配图。

**考虑过的替代**：
- AI 直接写库：效率高但错误率无门槛
- 完全人工：质量最高但 P6 旧数据迁移不可能完成

**后果**：
- 优势：错误有人兜底，AI 模型可独立升级
- 代价：录入速度受人工审查瓶颈影响；通过"同卷录入"模式批量接受可缓解

## ADR-006 · 答案显示模式 0-3

**日期**：2026-07-13
**状态**：已采纳
**阶段**：P0（设计）/ P2（实现）

**背景**：同一份试卷在不同场景需要不同呈现：
- 讲义：完整题干 + 解答 + 答案
- 练习册：题干 + 解答区留白，答案隐藏
- 考试卷：题干 + 固定空白，答案隐藏
- 测试卷：仅题干，完全不显示

**方案**：组卷时选 `answer_mode ∈ {0, 1, 2, 3}`，导出时按模式渲染。

**后果**：
- 优势：一份题库覆盖多场景
- 代价：导出模板需支持 4 种条件分支（增加渲染复杂度）

## ADR-007 · 新项目独立建设，不与 mathVault 耦合

**日期**：2026-07-13
**状态**：已采纳
**阶段**：P0

**背景**：mathVault 是基于 Obsidian wiki 的旧题库系统，Markdown 文件 + YAML frontmatter 存储。新系统以 SQLite 为唯一真相，wiki 退化为可重新生成的导出格式。

**方案**：完全独立的 `MathForge/` 目录，不读取 mathVault 任何文件。P6 阶段提供 `scripts/import_legacy.py` 做一次性迁移（从 wiki/ 读 → 写 DB），迁移后两系统完全解耦。

**后果**：
- 优势：旧系统可以保留为只读档案，不影响新系统迭代
- 代价：迁移脚本是一次性投入；字段映射在导入时确认后即固化
